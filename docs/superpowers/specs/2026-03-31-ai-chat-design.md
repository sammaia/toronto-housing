# AI Chat — Design Spec
**Date:** 2026-03-31
**Status:** Approved

---

## Overview

Add an AI-powered chat feature to Toronto Housing Insights. Users ask natural language questions about housing data; a Claude agent retrieves data via tools, streams the response token-by-token, and persists conversations in the database. Conversations survive page refresh and can be resumed at any time.

**Tech:** Anthropic SDK (Claude Haiku, model configurable via env), tool calling, Server-Sent Events, Prisma/PostgreSQL, NestJS, React.

---

## Architecture

```
Frontend                    Backend (NestJS)              Anthropic API
─────────                   ────────────────              ─────────────
ChatPage
  ├─ ConversationList  ──── GET  /chat/conversations
  │                    ──── POST /chat/conversations
  │                    ──── DELETE /chat/conversations/:id
  └─ ChatThread        ──── GET  /chat/conversations/:id
       └─ MessageInput ──── POST /chat/conversations/:id/messages (SSE)
                                     │
                              ChatService
                                     │── load history from DB
                                     │── claude.messages.stream()
                                     │       ▼ text_delta  → SSE text
                                     │       ▼ tool_use    → ToolService.run()
                                     │                     → SSE tool_start/end
                                     │       ▼ tool_result → Claude (loop)
                                     │       ▼ stop
                                     │── save assistant message to DB
                                     │── generate title (1st message only)
                                     └── SSE done
```

All endpoints are JWT-protected using the existing auth guard.
No existing modules are modified — chat is purely additive.

## Database Schema

```prisma
model Conversation {
  id        String    @id @default(cuid())
  userId    String
  title     String?
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt

  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  messages  Message[]
}

model Message {
  id             String      @id @default(cuid())
  conversationId String
  role           MessageRole
  content        String      @db.Text
  toolCalls      Json?
  createdAt      DateTime    @default(now())

  conversation   Conversation @relation(fields: [conversationId], references: [id], onDelete: Cascade)
}

enum MessageRole {
  USER
  ASSISTANT
}
```

Add to existing `User` model:
```prisma
conversations Conversation[]
```

**Notes:**
- `title` is nullable; generated by Claude after the first exchange (~5-word summary)
- `toolCalls` stores which tools were called and with what inputs (useful for debugging and optional UI display)
- Cascade deletes keep the DB clean when users or conversations are removed

### Endpoints

| Method | Route | Description |
|--------|-------|-------------|
| GET | `/chat/conversations` | List user's conversations (id, title, updatedAt) |
| POST | `/chat/conversations` | Create new conversation, returns `{ id }` |
| GET | `/chat/conversations/:id` | Load full message history |
| DELETE | `/chat/conversations/:id` | Delete conversation |
| POST | `/chat/conversations/:id/messages` | Send message — SSE response |

### Agentic Loop (`ChatService.sendMessage`)

```
1. Validate conversation belongs to authenticated user
2. Save USER message to DB
3. Load full conversation history from DB → format as Anthropic messages[]
4. Open claude.messages.stream({ model, system, messages, tools, max_tokens: 1024 })
5. Loop until stop_reason === 'end_turn':
   a. text_delta       → SSE { type: "text", content }
   b. tool_use block   → SSE { type: "tool_start", tool, input }
                       → ToolService.run(toolName, input)
                       → SSE { type: "tool_end", tool }
                       → append tool_result to messages
                       → call Claude again (new stream)
6. Save ASSISTANT message to DB with content + toolCalls JSON
7. If first message in conversation → generate title via separate Claude call (non-streaming, haiku)
8. SSE { type: "done", conversationId, title? }
```

## SSE Protocol

Events sent from backend to frontend during streaming:

```typescript
{ type: "text";       content: string  }  // text token(s)
{ type: "tool_start"; tool: string; input: Record<string, unknown> }
{ type: "tool_end";   tool: string     }
{ type: "done";       conversationId: string; title?: string }
{ type: "error";      message: string  }
```

Frontend reads via `fetch` + `ReadableStream` (not `EventSource`, since we need POST with body).

### Component Responsibilities

- **ConversationList** — grouped by date (Today / Yesterday / Older), `+ New chat` button, highlights active conversation, updates title live when `done` event arrives
- **ChatThread** — scrollable message list, auto-scrolls during streaming, shows "thinking" dots before first token
- **ChatMessage** — renders Markdown (react-markdown), user vs assistant styles consistent with app identity (amber accent for user, neutral for assistant)
- **ToolCallIndicator** — animated amber pill: *"Consultando dados de aluguel…"* shown between `tool_start` and `tool_end` events
- **MessageInput** — textarea, submit on `Cmd+Enter` or button, disabled + shows stop cursor during streaming
- **useChatStream** — custom hook encapsulating fetch + ReadableStream parsing, returns `{ messages, isStreaming, sendMessage }`

### State Management

Local state per `ChatPage` — no global store needed:
- `conversations: ConversationSummary[]` — list for sidebar
- `activeConversationId: string | null`
- `messages: Message[]` — current thread
- `isStreaming: boolean`

## Error Handling

| Scenario | Behavior |
|----------|----------|
| Anthropic API key missing | 500 on first message; log clear error server-side |
| Tool execution fails | Tool returns `{ error: "..." }` to Claude; Claude handles gracefully in response |
| SSE connection drops | Frontend shows inline error; user can retry |
| Conversation not found / wrong user | 404; frontend redirects to new conversation |
| Rate limit hit | 429 forwarded as SSE `error` event with user-friendly message |
